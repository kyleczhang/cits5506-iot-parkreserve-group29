# Pi-Side Notes for the Reservation Lifecycle Sync

This note is for the Raspberry Pi / gateway implementation. It explains the
MQTT contract the cloud backend actually enforces today, so the Pi code can
match it.

Related references:

- [backend-implementation-plan.md](./backend-implementation-plan.md)
- [asyncapi.yaml](./asyncapi.yaml)
- [topics.py](../../backend/app/mqtt/topics.py) — pydantic schemas with
  `extra="forbid"` — these reject any unknown field at validation time.

---

## 1. Why this contract exists

Two responsibilities are clearly split:

- The **Pi owns the physical bay state machine**. It merges sensor +
  LPR + cached reservation into a bay state, drives LEDs / buzzer, and
  emits state-change and event messages on transition.
- The **backend owns reservation business timing** — arrival grace,
  check-in grace, late-cancel cutoff, no-show, weak-conflict. The Pi
  does *not* run timers for those.

The Pi must not implement its own `ARRIVAL_GRACE_MINUTES` or
`CHECK_IN_GRACE_MINUTES` countdowns for reservation business decisions.
Those grace windows belong to the backend and are evaluated by
[`app/jobs/reconcile_reservations.py`](../../backend/app/jobs/reconcile_reservations.py).

## 2. MQTT contract

Topic prefix is configurable on the backend via `MQTT_TOPIC_PREFIX`
(default `cloud`). All examples below assume that default.

### 2.1 Topics

| Direction | Topic | Purpose |
|-----------|-------|---------|
| Pi → backend | `cloud/bay/<code>/state` | Bay-state mirror + last distance reading |
| Pi → backend | `cloud/bay/<code>/event` | Pi-originated state-machine events |
| Pi → backend | `cloud/system/heartbeat` | Pi heartbeat (~10 s) |
| Backend → Pi | `cloud/bay/<code>/reservation` | Reservation commands (+ bound plates) |
| Backend → Pi | `cloud/system/resync` | "Replay your current bay state" |

QoS 1 on every business topic. No retained messages, no LWT today —
resync on reconnect is the recovery mechanism.

### 2.2 Pi → backend `event` is narrower than the union of all events

The Pi may publish **only** these `event` values on `cloud/bay/<code>/event`:

```text
sensor_online
sensor_offline
auto_check_in
pending_check_in
check_in_confirmed
conflict_strong
```

The Pi must **not** publish:

- `conflict_weak`
- `no_show`

Those are backend-synthesised internal events, generated by the
reservation safety-net sweeper after the relevant grace expires.
[`app/mqtt/topics.py`](../../backend/app/mqtt/topics.py) splits this into
two pydantic schemas (`PiInboundEventPayload` and `InternalEventPayload`)
and the inbound MQTT path only accepts the former — sending `conflict_weak`
or `no_show` on `cloud/bay/<code>/event` is rejected at schema validation
and logged as `mqtt.invalid_event`.

### 2.3 Backend → Pi reservation command actions

Allowed actions on `cloud/bay/<code>/reservation`:

```text
create
cancel
check_in
update_plates
release
expire_check_in
```

Every payload carries the user's *current* `bound_plates` list so the Pi's
LPR matcher always has the freshest set — there is no separate "plates"
topic. The action `update_plates` exists for the case where a plate was
added or removed during a live reservation and the rest of the fields
haven't changed.

### 2.4 `release` requires `reason`

```text
reason = no_show | completed | abandoned | admin_override
```

`reason` is **required** when `action="release"` and **forbidden**
otherwise. This is enforced by
[`ReservationCommand.validate_reason`](../../backend/app/mqtt/topics.py).

### 2.5 Payload schemas

`StatePayload`:

```json
{ "state": "available",
  "last_distance_cm": 120.5,
  "ts": "2026-05-01T10:00:00Z",
  "event_id": "8c0c..." }   // optional; used for idempotency on bay_events
```

`PiInboundEventPayload`:

```json
{ "event": "auto_check_in",
  "ts": "2026-05-01T10:00:00Z",
  "event_id": "...",
  "reservation_id": "...",            // optional
  "recognised_plate": "ABC123",       // required for auto_check_in / conflict_strong
  "lpr_confidence": 0.91 }            // optional, 0.0 ≤ x ≤ 1.0
```

`ReservationCommand` (Pi receives):

```json
{ "action": "create",
  "reservation_id": "...",
  "user_id": "...",
  "bound_plates": ["ABC123", "XYZ789"],
  "expected_arrival_time": "2026-05-01T10:30:00Z",
  "reason": null,
  "ts": "2026-05-01T10:00:00Z" }
```

All schemas use `model_config = ConfigDict(extra="forbid")`: extra fields
the Pi adds will be rejected by the backend. Keep payloads strictly to
the documented surface.

## 3. Backend → Pi command semantics

### 3.1 `create`

A new reservation just bound to this bay. Payload includes
`expected_arrival_time`, `user_id`, `reservation_id`, and the
`bound_plates` list. Pi cache: store the reservation; expect a
vehicle around the arrival time.

The Pi should NOT decide no-show timing — even after arrival passes, wait
for either:

- a vehicle arrival, or
- a `release` from the backend.

### 3.2 `cancel`

The holder cancelled (cleanly or late — the Pi does not need to know
which). Pi cache: remove the reservation. The bay should fall back to the
sensor-derived state (`available` if empty, `occupied` if a vehicle is
sitting there).

The backend also publishes `cancel` (not `release`) on the holder's
voluntary cancel; `release` is reserved for situations where there is a
specific business reason for the Pi to know.

> Implementation note: the backend currently publishes `cancel` for both
> the clean-cancel and late-cancel paths. If we later need the Pi to
> distinguish, we will add the case to `release(reason=...)` rather than
> overloading `cancel`.

### 3.3 `check_in`

A user-initiated check-in (QR or manual) succeeded server-side, OR the
backend is restoring a `CHECKED_IN` reservation after a strong-conflict
clear. Pi behaviour:

- mark the reservation as checked-in;
- move local bay state to `reserved_checked_in`;
- if a weak-conflict alert was active, clear it.

### 3.4 `update_plates`

Same reservation; only the plate list changed (user added or removed a
bound plate while a live reservation was running). Pi behaviour: swap in
the new `bound_plates` for this reservation's cache entry. No state
machine transition.

### 3.5 `release`

The reservation is no longer active from the backend's point of view.
Pi behaviour:

1. Remove the local reservation cache for this bay / reservation.
2. Clear reservation-related warning state.
3. Immediately recompute bay state from the **current** sensor reading.

Recompute rule:

- sensor empty → `available`
- sensor occupied → `conflict` (casual / unknown vehicle still there)

Don't wait for the next sensor edge — the cloud has already finalised
the lifecycle.

`release.reason` values:

| Reason | When the backend sends it |
|--------|---------------------------|
| `no_show` | Sweeper synthesised a `no_show`. In practice the bay was empty, so recompute should normally land on `available`. |
| `completed` | The backend inferred `COMPLETED` from a `reserved_checked_in → available` transition (the holder drove off). |
| `abandoned` | The backend inferred `COMPLETED` from a `pending_check_in → available` transition (vehicle entered then left without a real check-in), or the reservation finished from `IN_CONFLICT`. |
| `admin_override` | Admin used the admin termination path, OR the holder voluntarily cancelled while a strong conflict was open on their bay. |

### 3.6 `expire_check_in`

The weak-conflict grace timer expired (backend-owned). The backend has
captured a `weak_conflict` penalty and released the deposit remainder
on the cloud side. Pi behaviour:

1. **Keep** the local reservation cache (the user may still arrive late
   and manually check in).
2. Move local bay state to `conflict`.
3. Enable weak-conflict alert behaviour locally (red blinking LED +
   buzzer, per proposal §5.4).
4. Keep accepting a later `check_in` command for the same reservation —
   if the user does manage a manual check-in, the backend will resolve
   the weak conflict as `user_arrived_and_checked_in` and publish
   `action=check_in`. Clear the alert and move to `reserved_checked_in`.

## 4. Required Pi → backend behaviour

### 4.1 What the Pi publishes

- On every state change: `cloud/bay/<code>/state` with the new state, the
  last distance reading, and a fresh `event_id` UUID.
- On every state-machine event: `cloud/bay/<code>/event` with one of the
  six allowed `event` values.
- Roughly every 10 s: `cloud/system/heartbeat` with `pi_id` and `ts`.

`event_id` should be a UUID v4 freshly generated for each event. The
backend uses it as the idempotency key for `bay_events.source_event_id`,
`conflicts.source_event_id`, and (for `conflict_weak` / `no_show`)
`payments.source_event_id`. Reusing an `event_id` across two semantically
different events would suppress one of them.

### 4.2 Resync handling

When the Pi receives `cloud/system/resync` (payload `{"request":"replay"}`):

- For every bay it owns, publish a current `cloud/bay/<code>/state`
  message immediately.
- Optionally republish recent events the Pi believes the backend may
  have missed. The backend's `bay_events.source_event_id` unique
  constraint makes replays no-ops as long as the same `event_id` is used.

The backend's inbound MQTT client always publishes `resync` on `on_connect`,
including reconnects.

### 4.3 Strong-conflict evidence image

After publishing `conflict_strong` on MQTT, the Pi additionally POSTs the
JPEG to the backend over HTTPS:

```
POST /api/v1/internal/conflicts/evidence
Authorization: Bearer <EVIDENCE_UPLOAD_TOKEN>
Content-Type: multipart/form-data

bay_code=A1
source_event_id=<same UUID as the MQTT event_id>
recognised_plate=ABC123
image=@/tmp/conflict.jpg
```

The backend's `EVIDENCE_UPLOAD_TOKEN` is the shared bearer; if unset,
authentication is skipped (dev mode only). The two halves (MQTT event +
HTTPS upload) may arrive in either order — both upsert the same
`conflicts` row keyed on `source_event_id`.

## 5. Pi-side state machine sequences

### 5.1 Normal happy path (auto check-in)

1. Backend publishes `reservation/create`.
2. Pi: cache reservation; state → `reserved`.
3. Vehicle arrives, sensor goes occupied. Pi runs LPR.
4. LPR matches a bound plate. Pi:
   - publishes `event=auto_check_in` with `recognised_plate`,
   - state → `reserved_checked_in`.
5. Vehicle drives away. Sensor empty.
6. Pi: state → `available`. Pi publishes `state=available`.
7. Backend infers `COMPLETED`, publishes `release(reason=completed)`.
8. Pi: drop reservation cache.

### 5.2 Pending → manual check-in

1. `reservation/create` received.
2. Vehicle arrives. LPR does not auto-match (low confidence / wrong plate
   shape / no plate visible).
3. Pi: publishes `event=pending_check_in`; state → `pending_check_in`.
4. User taps "I'm here" in the dashboard or scans the bay QR code.
5. Backend calls the reservation service, which:
   - flips reservation → `CHECKED_IN`,
   - publishes `reservation/check_in`.
6. Pi: clear pending state; state → `reserved_checked_in`.
7. (Vehicle leaves → as in §5.1 steps 5–8.)

### 5.3 Weak-conflict timeout path

1. `event=pending_check_in` published (as §5.2 step 3).
2. `CHECK_IN_GRACE_MINUTES` later, the backend sweeper synthesises
   `conflict_weak`. Backend publishes `expire_check_in` to the Pi and
   captures a `weak_conflict` penalty.
3. Pi: state → `conflict`; raise alert; keep reservation cache.
4. If the user later manages a manual check-in via the API, the backend
   resolves the weak conflict and publishes `check_in`. Pi clears alert
   and moves to `reserved_checked_in`.
5. Otherwise the vehicle eventually leaves; bay returns to `available`;
   reservation completes as `COMPLETED` (with `release(reason=abandoned)`
   or `release(reason=completed)` depending on whether the user had
   recovered through check-in first).

### 5.4 No-show path

1. `reservation/create` received.
2. No vehicle arrives before `expected_arrival_time + ARRIVAL_GRACE_MINUTES`.
3. Sweeper synthesises `no_show`. Backend captures `no_show` penalty and
   publishes `release(reason=no_show)`.
4. Pi: drop reservation cache; recompute. Bay should normally be
   `available`.

### 5.5 Strong-conflict path

1. `reservation/create` received.
2. Wrong vehicle arrives. LPR returns a plate that is NOT in the
   reservation's bound list.
3. Pi:
   - publishes `event=conflict_strong` with `recognised_plate` and
     `lpr_confidence`,
   - HTTP-POSTs the JPEG to `/internal/conflicts/evidence` keyed on
     the same `event_id`,
   - state → `conflict`; raise alert.
4. The backend does NOT refund automatically. The reservation row is
   preserved (status rolls back to `ACTIVE` if it was `PENDING_CHECK_IN`).
5. Two terminations:
   - **The wrong vehicle drives away.** Pi: state → `available`
     (or whatever the sensor reads). Backend sees the
     `conflict → available` transition, resolves the conflict as
     `vehicle_left`, resumes the reservation, and republishes
     `reservation/create` (or `reservation/check_in`) to the Pi.
   - **An admin (or the holder) terminates the reservation.** The
     backend issues a full refund and publishes
     `release(reason=admin_override)`. Pi: drop reservation cache;
     recompute. The wrong vehicle is still likely there, so the
     recompute will land on `conflict` until it leaves.

## 6. What the Pi does NOT need to do

- Track `ARRIVAL_GRACE_MINUTES` countdowns for reservation business decisions.
- Track `CHECK_IN_GRACE_MINUTES` countdowns for reservation business decisions.
- Decide or publish `no_show`.
- Decide or publish `conflict_weak`.
- Manage payments, deposits, or penalties (the Pi is payment-agnostic).
- Issue any reservation-state changes that the backend hasn't asked for
  via `cloud/bay/<code>/reservation`.

## 7. Suggested Pi implementation checklist

1. Reservation command parser accepts all six actions, including
   `release` (with required `reason`) and `expire_check_in`.
2. Reject any reservation command whose `reservation_id` doesn't match
   the bay's currently cached reservation (defensive against stale
   broker buffers).
3. Local recompute on `release` is **immediate** (don't wait for the next
   sensor edge).
4. Cache stays on `expire_check_in`; a subsequent `check_in` clears the
   alert.
5. Each Pi-published event carries a unique `event_id` (UUID v4).
6. HTTPS evidence upload uses the SAME `event_id` as the corresponding
   MQTT `conflict_strong` event.
7. On (re)connect / on receipt of `cloud/system/resync`, replay current
   bay state for every owned bay.
8. Heartbeat every ~10 s on `cloud/system/heartbeat`.
9. Outbound LPR runs ONLY when the cached state is `reserved` (privacy
   policy — no LPR for casual parking, proposal §5.5).

## 8. Integration tests worth keeping

| Scenario | Verify |
|----------|--------|
| Weak-conflict late check-in | `pending_check_in → expire_check_in → conflict → check_in → reserved_checked_in` |
| No-show cleanup | `no Pi vehicle → release(no_show) → cache cleared → available` |
| Normal completion | `auto_check_in → reserved_checked_in → vehicle leaves → state=available → release(completed) → cache cleared` |
| Abandoned cleanup | `pending_check_in → vehicle leaves before check-in → release(abandoned) → cache cleared` |
| Strong-conflict drive-away (preserve) | `conflict_strong → wrong vehicle leaves → state=available → backend resolves vehicle_left → reservation/create republished` |
| Strong-conflict admin terminate | `conflict_strong → admin terminate → release(admin_override) → cache cleared` |
| Resync | Backend sends `cloud/system/resync` on reconnect; Pi replays every owned bay's state |

The backend already has matching cases under
[backend/tests/](../../backend/tests/):
`test_event_handlers.py`, `test_strong_conflict_recovery.py`,
`test_reconcile_job.py`, `test_resilience_reconnect.py`.

## 9. Known limitation

If the Pi is offline when the backend tries to send `release` or
`expire_check_in`, delivery is not guaranteed by a persistent MQTT
session in the current dev setup (HiveMQ Cloud config has not yet
enabled persistent sessions for the backend client). For now:

- the Pi still supports `cloud/system/resync` and replays current state
  on reconnect;
- the backend re-publishes the latest reservation command on certain
  transitions (e.g. strong-conflict restore), so reservation cache will
  resync;
- timeout-driven payment transitions (no_show / conflict_weak penalty
  captures + releases) are owned entirely by the backend and don't
  depend on Pi acknowledgement.

That is a current system limitation, not a Pi-only bug.
